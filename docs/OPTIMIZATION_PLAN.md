## 一、项目现状分析

### 1.1 系统架构总览

当前项目为 ROS2 Humble 多节点架构，包含三套独立方案共享下游通路：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        HNURM Radar 2026                           │
├─────────────┬─────────────────────┬─────────────────────────────────┤
│ 方案一(纯相机) │ 方案二(相机+雷达)    │ 空中方案(纯激光雷达)             │
│             │                     │                                 │
│ camera_     │ detector_node       │ air_target_node                 │
│ detector    │ (YOLO 三阶段)        │ (背景减除+DBSCAN+卡尔曼)        │
│ (YOLO+透视) │         ↓            │         ↓                       │
│      ↓      │ radar_node          │  TF坐标变换(livox→map)           │
│ Homography  │ (点云投影+聚类)       │         ↓                       │
│      ↓      │         ↓            │                                 │
├─────────────┴─────────────────────┴─────────────────────────────────┤
│                    共享下游通路                                       │
│  /location → ekf_node(EKF滤波) → /ekf_location_filtered            │
│                    ↓                          ↓                      │
│           display_panel(小地图)      judge_messager(裁判系统串口)      │
└─────────────────────────────────────────────────────────────────────┘
```

**硬件平台**：
- 激光雷达：Livox HAP（非重复扫描固态雷达）
- 工业相机：海康威视 MV-CS016-10UMUC / MV-CS060-10UMUC-PRO
- 运算平台：220V 市电供电，最大 750W
- 裁判系统：有线串口接入场地主控模块

**技术栈**：
- ROS2 Humble + Python (主要) + C++ (Quatro/Registration)
- Ultralytics YOLO（内嵌修改版，含自定义 backbone：EfficientFormerV2、DCNv4 等）
- Open3D (点云处理)、OpenCV (图像处理)
- ByteTrack (多目标跟踪)
- ICP/Quatro (点云配准)

### 1.2 三种方案现状

| 维度 | 方案一（纯相机） | 方案二（相机+雷达） | 空中方案（纯雷达） |
|------|:---------------:|:------------------:|:-----------------:|
| **成熟度** | ★★★☆☆ | ★★★★☆（主力方案） | ★★☆☆☆ |
| **目标类型** | 地面机器人 | 地面机器人 | 空中无人机 |
| **硬件需求** | 仅相机 | 相机 + Livox HAP | 仅 Livox HAP |
| **定位方式** | 透视变换 2D→2D | 点云投影+聚类 2D→3D | 背景减除+DBSCAN 3D |
| **检测模型** | YOLO 三阶段级联 | YOLO 三阶段级联 | 无（纯点云聚类） |
| **当前问题** | 标定依赖高、高地精度差 | 启动流程复杂、需手动 ICP | 仅测试过单台无人机 |

**YOLO 三阶段级联检测流程**（方案一/二共用）：

```
Stage 1: 全图目标检测 (stage_one.pt, imgsz=1280, conf=0.1)
         + ByteTrack 多目标追踪
              ↓ 裁剪 ROI
Stage 2: 装甲板颜色+编号分类 (stage_two.pt, imgsz=256, conf=0.6)
         输出 12 类: B1-B5,B7,R1-R5,R7
              ↓
Stage 3: 灰色装甲板专用分类 (stage_three.pt, imgsz=256, conf=0.5)
         输出 6 类灰色装甲板
              ↓
投票机制: Track_value 加权累加 (0.5+conf*0.5), 每 90 帧衰减 1/10
```

### 1.3 雷达评分机制与开发优先级推导

根据规则手册 V1.4.0，雷达竞技评分公式为：

```
时均雷达表现分数 = (累计准确标记时间 × 1 + 反制无人机次数 × 20 + 成功解析信息波难度等级 × 200)
                  / 雷达所属队伍正式比赛总时长(秒)
```

**收益分析**（假设一场比赛 7 分钟 = 420 秒）：

| 评分要素 | 权重 | 理论最大贡献 | 开发难度 | **投入产出比** |
|----------|------|-------------|---------|:----------:|
| 准确标记时间 | ×1 | ~420 分（满标记） | 中 | ★★★★★ |
| 反制无人机 | ×20 | ~60 分（3次反制） | 高 | ★★★☆☆ |
| 解析信息波 | ×200 | ~400+ 分（≥2级） | 极高 | ★★★★☆ |

**结论**：
1. **准确标记**是基础分数来源，投入产出比最高 — 优先保证地面机器人识别准确率和坐标精度
2. **信息波解析**单次收益最高 — 中期重点攻坚，需要 SDR 硬件和信号处理能力
3. **反制无人机**需要激光云台硬件 + 空中定位 — 长期推进，当前先确保空中定位可用

### 1.4 当前技术债与已知问题

| 类别 | 问题描述 | 影响 |
|------|---------|------|
| **模型** | Ultralytics 版本过旧（内嵌修改版），无法使用 YOLOv11 等新特性 | 检测精度受限 |
| **模型** | 三阶段串行推理延迟高，Stage 2/3 独立训练数据少 | FPS 受限 |
| **配置** | `configs/` 中存在废弃配置（`archive/`），配置项过多且分散 | 维护成本高 |
| **代码** | `judge_messager.py` 977 行单文件，通信逻辑简陋 | 可维护性差 |
| **代码** | `detector_node.py` 中 `Track_value` 固定 10000 长度数组 | 内存浪费 |
| **代码** | `ekf_node.py` 路径硬推导（`os.path.join(__file__, '..', '..', '...')`） | 易出错 |
| **工程** | `.gitignore` 将整个 `docs/` 排除，文档不入版本控制 | 知识流失 |
| **工程** | `.pt` 权重文件直接入库（未配 Git LFS），`.engine` 被排除 | 仓库体积膨胀 |
| **启动** | 方案二启动需 3 个终端 + 手动 RViz 给 2D Pose Estimate | 操作繁琐 |
| **功能** | 裁判系统仅实现坐标上报和双倍易伤，缺少多机通信、信息波等 | 功能不全 |
| **功能** | 无激光反制空中机器人模块 | 评分项缺失 |
| **功能** | 无电磁波信息波解析模块 | 评分项缺失 |

## 三、阶段一：工程基建与数据治理

> **目标**：建立可持续迭代的工程基础，解决历史技术债，为后续算法优化提供数据和工具支撑。
>
> **预计周期**：4 周 | **优先级**：🔴 最高

### 3.1 历史数据整理与管理

#### 3.1.1 问题现状

- 比赛视频、点云录制分散存储，缺乏统一命名和索引
- 标注数据（如装甲板检测数据集）无版本管理
- `data/` 目录混杂背景模型、点云地图、临时文件
- `record/` 在 `.gitignore` 中被排除，录制数据易丢失

#### 3.1.2 改进方案

**A. 数据目录规范化**

```
data/
├── datasets/                    # 训练数据集（Git LFS 或外部存储）
│   ├── armor_det_v1/            # 装甲板检测数据集
│   │   ├── images/
│   │   ├── labels/
│   │   └── dataset.yaml
│   ├── armor_cls_v1/            # 装甲板分类数据集
│   └── README.md                # 数据集说明
├── maps/                        # 场地地图（从 map/ 迁移）
│   ├── competition_2026/
│   └── lab/
├── pointclouds/                 # 点云数据
│   ├── background/              # 背景模型
│   └── calibration/             # 标定用点云
├── recordings/                  # 比赛/训练录制
│   ├── YYYYMMDD_event_round/    # 按日期-事件-轮次命名
│   │   ├── video/
│   │   ├── rosbag/
│   │   └── metadata.yaml        # 录制元信息
│   └── index.yaml               # 录制索引
└── weights/                     # 模型权重（从 weights/ 迁移）
    ├── stage_one.pt
    ├── stage_two.pt
    ├── stage_three.pt
    └── README.md                # 模型版本说明
```

**B. 数据版本管理策略**

| 数据类型 | 管理方式 | 说明 |
|---------|---------|------|
| 训练数据集 | Git LFS + DVC | 大文件不入仓库主体，用 DVC 追踪版本 |
| `.pt` 权重 | Git LFS | 当前直接入库导致仓库膨胀，需迁移 |
| `.engine` 文件 | 不入库 | TensorRT 引擎与硬件绑定，保持在 `.gitignore` |
| 背景模型 `.pcd` | Git LFS | 与场景绑定，需版本管理 |
| 录制数据 | 外部 NAS/对象存储 | 通过 `index.yaml` 索引，不入仓库 |
| 地图图片 | 普通 Git | 文件较小，直接入库 |

**C. 标注流程标准化**

1. 使用 [Roboflow](https://roboflow.com/) 或 [CVAT](https://www.cvat.ai/) 统一标注平台
2. 标注规范文档：定义装甲板标注的锚框大小、遮挡处理、模糊阈值
3. 数据增强策略：运动模糊、光照变化、装甲板视角旋转
4. 数据集划分：训练集 70% / 验证集 20% / 测试集 10%，确保场景多样性

**B. Ultralytics 依赖管理**

当前项目中内嵌了一个修改版的 Ultralytics（含自定义 backbone：LSKNet、SwinTransformer、TransNext 等）。建议：

1. **短期**：保持内嵌但记录修改点，创建 `ultralytics_patches.md`
2. **中期**：将自定义模块提取为独立插件，基于官方 Ultralytics 的插件机制注册
3. **长期**：评估是否迁移到 YOLOv11/YOLO-World，利用开放词汇检测降低标注成本

**C. 模型卡（Model Card）制度**

每个模型版本必须附带：
- 训练数据集版本和规模
- 训练超参数（学习率、epoch、batch size 等）
- 验证集指标（mAP@0.5、mAP@0.5:0.95）
- 推理性能（FPS@TensorRT FP16、GPU 内存占用）
- 已知局限性

#### 4.1.1 当前方案分析

**三阶段串行检测管线**：

### 4.2 点云背景抠除优化

#### 4.2.1 当前方案分析

**现有实现**（`air_target_node.py`）：

```
原始点云 → TF 变换到 map 坐标系 → ROI 裁剪 → 背景抠除 → Z 轴压缩 → DBSCAN 聚类 → 卡尔曼跟踪
```

**背景抠除原理**：
1. 预先采集空场场景的点云，保存为 `background.pcd`
2. 将背景点云构建 KDTree（`AirBackgroundConfig.method = "kdtree"`）
3. 运行时对每个点查询最近邻距离，距离 > 阈值则判定为前景
4. 前景点参与后续聚类和跟踪

**当前配置**（`main_config.yaml`）：

```yaml
background:
  method: kdtree                 # 仅支持 kdtree
  kdtree_radius: 0.2             # 最近邻距离阈值（米）
  min_foreground_points: 5       # 最少前景点数
  voxel_size: 0.08               # 体素降采样大小
```

**已知问题**：

| 问题 | 影响 |
|------|------|
| 静态背景模型无法适应光照、温度导致的点云漂移 | 误检率高 |
| KDTree 查询对大规模点云计算量大 | 延迟增加 |
| 背景采集需要人工清场，操作不便 | 部署效率低 |
| Livox HAP 非重复扫描导致单帧密度不均 | 近处密、远处疏 |
| 无动态障碍物（如人员走动）的处理 | 误检 |

#### 4.2.2 改进方案

**方案 A：改进 KDTree 方案（短期）**

1. **自适应阈值**：根据点到 LiDAR 的距离动态调整背景匹配阈值
   ```python
   # 近处阈值小（密度高，精度高），远处阈值大（密度低，允许更大偏差）
   adaptive_radius = base_radius * (1 + 0.1 * distance_to_lidar)
   ```
2. **时间衰减背景模型**：在线更新背景，新观测逐渐融入
   ```python
   # 每 N 帧用当前"无目标区域"更新背景
   background = alpha * background + (1 - alpha) * current_frame
   ```
3. **多分辨率体素滤波**：近处细粒度、远处粗粒度

**方案 B：基于八叉树的背景建模（中期）**

```
优势：
- 空间自适应分辨率
- 查询效率 O(log N)
- 天然支持动态更新

实现：
- 使用 Open3D 的 Octree 或 octomap
- 每个体素记录占据概率
- 占据概率高于阈值的体素为背景
```

**方案 C：深度学习点云分割（长期探索）**

| 模型 | 特点 | 适用性 |
|------|------|--------|
| PointPillars | 快速，适合稀疏点云 | ⭐⭐⭐ |
| CenterPoint | 中心点检测，抗遮挡 | ⭐⭐ |
| 3D-MiniNet | 轻量级，适合嵌入式 | ⭐⭐⭐ |

### 4.3 EKF 轨迹预测优化

#### 4.3.1 当前方案分析

**现有实现**（`ekf_node.py`）：

- 维护 7 个独立 EKF 滤波器：`robot_1` ~ `robot_5`（地面）、`sentry`（哨兵）、`air`（空中）
- 状态向量：`[x, y, z, vx, vy, vz]`（位置 + 速度）
- 观测输入：来自检测节点的 `/location` 话题（`Float32MultiArray`）
- 输出：`/ekf_location_filtered` 话题
- 更新频率：由 `timer_callback` 控制

**已知问题**：

| 问题 | 详情 |
|------|------|
| 滤波器参数固定 | 过程噪声 Q、观测噪声 R 未区分运动状态 |
| 无数据关联 | 检测 ID 跳变时 EKF 无法正确关联目标 |
| 无遮挡处理 | 目标被遮挡时 EKF 无观测输入，状态快速发散 |
| 无运动模型切换 | 匀速模型对加速/转弯场景适应性差 |
| 独立滤波 | 7 个滤波器互不通信，无法利用全局约束 |

#### 4.3.2 改进方案

**A. 参数自适应（短期）**

```python
# 根据检测置信度调整观测噪声 R
def adaptive_R(confidence):
    """置信度高 → R 小（信任观测）; 置信度低 → R 大（信任预测）"""
    base_R = np.diag([0.5, 0.5, 0.5])
    return base_R / (confidence + 0.1)

# 根据速度大小调整过程噪声 Q
def adaptive_Q(velocity):
    """速度大 → Q 大（允许更大状态变化）"""
    speed = np.linalg.norm(velocity)
    base_Q = np.diag([0.1, 0.1, 0.1, 1.0, 1.0, 1.0])
    return base_Q * (1 + 0.5 * speed)
```

**B. 交互多模型（IMM）滤波器（中期）**

```
模型 1: 静止模型 (Nearly Constant Position)
模型 2: 匀速模型 (Nearly Constant Velocity)
模型 3: 匀加速/转弯模型 (Coordinated Turn)

IMM 算法在三个模型间切换，根据残差自动选择最优模型
适合处理机器人 "移动—停止—转向" 的运动模式
```

**C. 遮挡预测机制**

```python
class OcclusionAwareEKF:
    def update(self, observation=None):
        if observation is not None:
            self.consecutive_miss = 0
            self.ekf.update(observation)
        else:
            self.consecutive_miss += 1
            if self.consecutive_miss < MAX_MISS:
                # 仅执行预测步骤，保持惯性运动
                self.ekf.predict()
            else:
                # 超过阈值，标记目标丢失
                self.mark_lost()
```

**D. 全局约束**

- 地图边界约束：预测位置不超出场地范围（28m × 15m）
- 互斥约束：不同 ID 的目标不应重叠
- 区域先验：己方半场出现敌方目标的概率低，可用贝叶斯先验调整

### 5.1 激光反制空中机器人模块

#### 5.1.1 规则解读

根据 [RADAR_RULES_REFERENCE.md](../docs/rules/RADAR_RULES_REFERENCE.md)：

- **反制机制**：雷达站的激光检测模块被空中机器人照射时，空中机器人的激光检测模块面积会递减
- **被反制进度**：雷达站可以通过激光发射装置照射空中机器人的激光检测模块
- **被反制效果**：空中机器人被反制后，其激光检测模块的有效面积缩小，降低空中机器人的反制能力
- **面积递减规则**：每次成功反制后，空中机器人的激光检测模块面积按比例缩小
- **评分贡献**：`UAV_counter × 20` 计入雷达评分公式

### 5.2 电磁波信息波解析模块

#### 5.2.1 规则解读

根据规则手册：

- **频段**：433 MHz
- **调制方式**：2-GFSK（Gaussian Frequency Shift Keying）
- **信息波特征**：
  - 前导码 + 同步字 + 数据帧
  - 数据帧包含 6 字符密钥
  - 信息波与干扰波同时存在，需区分
- **难度等级**：初级/中级/高级，密钥长度和干扰程度递增
- **解析成功效果**：`signal_wave_level × 200` 计入评分公式
- **评分权重最高**：200 分/级，是所有评分项中单项权重最大的

#### 5.2.2 系统设计

┌───────────────────────────────────────────────────────────────┐

│                                                               │

│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐  │
│  │ 433MHz   │──→│ 2-GFSK   │──→│ 帧解析   │──→│ 密钥上报  │  │
│  │ SDR/     │   │ 解调     │   │ CRC 校验 │   │          │  │
│  │ 射频模块  │   │          │   │ 密钥提取 │   │          │  │
│  └──────────┘   └──────────┘   └──────────┘   └──────────┘  │
│       │               │              │                        │

│  └──────────┘   └──────────┘   └──────────┘                  │
└───────────────────────────────────────────────────────────────┘

#### 5.2.3 技术方案选择

| 方案 | 硬件 | 优缺点 |
|------|------|--------|
| **SDR（软件定义无线电）** | RTL-SDR / HackRF | 灵活性高，可用 GNURadio 快速开发；成本低 |
| **专用 433MHz 收发模块** | CC1101 / SI4463 | 硬件解调，延迟低；但灵活性差 |
| **FPGA + ADC** | 自研方案 | 性能最优但开发成本高 |

> 💡 **推荐**：使用 RTL-SDR + Python（`pyrtlsdr`）快速原型验证，确认可行后考虑是否需要专用硬件。

#### 5.2.4 2-GFSK 解调关键参数

根据规则附录：

```
载波频率: 433 MHz
数据速率: 待确认（通常 9.6 kbps ~ 38.4 kbps）
频偏: 待确认（通常 ±20 kHz ~ ±50 kHz）
BT 积: 0.5（高斯滤波器带宽时间积）

解调流程:

2. 频率鉴别器（FM 解调）
3. 符号同步（Gardner / Mueller-Müller）
4. 比特判决

6. 数据提取 + CRC 校验
```

### 5.3 VLA 决策模型集成

#### 5.3.1 目标定义

VLA（Vision-Language-Action）决策模型的目标是提供**战术级辅助决策**，而非替代人工操控。核心功能：

1. **双倍易伤时机建议**：根据当前战场态势（敌方血量、位置分布、剩余时间）推荐触发双倍易伤的最佳时机
2. **标记优先级排序**：多个敌方目标同时出现时，推荐优先标记哪个目标
3. **异常态势告警**：检测敌方异常集结、偷袭路线等

#### 5.3.2 技术路线

**短期方案：规则引擎**

```python
class TacticalAdvisor:
    """基于规则的战术建议器"""
    
    def should_activate_double_vulnerability(self, state: BattleState) -> bool:
        """判断是否触发双倍易伤"""
        # 规则 1: 己方英雄/步兵正在集火某目标且该目标标记进度 P > 100
        if any(p > 100 for p in state.enemy_marked_progress):
            # 规则 2: 剩余双倍易伤次数 > 0
            if state.remaining_double_times > 0:
                # 规则 3: 优先在对方血量厚的目标上使用
                target = max(state.enemies, key=lambda e: e.hp)
                if target.hp > 200:
                    return True
        return False
    
    def prioritize_targets(self, enemies: List[Enemy]) -> List[Enemy]:
        """目标优先级排序"""
        return sorted(enemies, key=lambda e: (
            -e.threat_level,      # 威胁度高的优先
            -e.marked_progress,   # 标记进度高的优先（容易触发易伤）
            e.hp,                 # 血量低的优先（容易击杀）
        ))
```

```python
from dataclasses import dataclass
import struct

@dataclass
class FrameHeader:
    sof: int = 0xA5
    data_length: int = 0
    seq: int = 0
    crc8: int = 0
    
    FORMAT = '<BHBB'  # 小端: uint8 + uint16 + uint8 + uint8
    SIZE = struct.calcsize(FORMAT)
    
    @classmethod
    def from_bytes(cls, data: bytes) -> 'FrameHeader':
        sof, length, seq, crc8 = struct.unpack(cls.FORMAT, data[:cls.SIZE])
        return cls(sof=sof, data_length=length, seq=seq, crc8=crc8)

@dataclass
class MapRobotData:
    """0x0305 雷达标记坐标数据"""
    target_robot_id: int
    target_position_x: float
    target_position_y: float
    
    FORMAT = '<Hff'
    SIZE = struct.calcsize(FORMAT)
    
    def to_bytes(self) -> bytes:
        return struct.pack(self.FORMAT,
            self.target_robot_id,
            self.target_position_x,
            self.target_position_y)
```

### 6.3 空中机器人定位方案重构

#### 6.3.1 当前问题

**现有方案**（`air_target_node.py`）：

- 完全依赖单台 LiDAR 的点云背景抠除
- ICP/Quatro 配准需要手动在 RViz 中给 2D Pose Estimate
- 空中目标点云稀疏，DBSCAN 聚类不稳定
- 无视觉辅助，纯点云检测对小型无人机困难

#### 6.3.2 改进方案

**A. 多传感器融合定位（推荐）**

```
相机检测                LiDAR 点云
(2D 框 + 类别)          (3D 背景抠除)
      │                      │
      ▼                      ▼
┌──────────┐          ┌──────────┐
│ 2D→射线  │          │ 3D 聚类  │
│ 投影     │          │ 候选     │
└──────────┘          └──────────┘
      │                      │
      └──────────┬───────────┘
                 ▼
          ┌──────────┐
          │ 射线-点云 │
          │ 关联匹配  │
          └──────────┘
                 │
                 ▼
          3D 位置估计
          (相机提供方向，LiDAR 提供深度)
```

**B. 自动化配准（消除手动 2D Pose Estimate）**

| 方案 | 说明 | 可行性 |
|------|------|--------|
| 全局定位器 | 使用 ScanContext / RING 等全局描述子进行初始配准 | ⭐⭐⭐ |
| 多点标记 | 预设多个已知位置，自动匹配 | ⭐⭐ |
| GPS 辅助 | 用 GPS/RTK 提供初始位姿 | ⭐（室内场无信号） |
| 预存位姿 | 雷达基座固定，预先标定好保存 | ⭐⭐⭐⭐ |

> 💡 **最实用方案**：雷达基座在场地上位置固定（规则规定），预先标定一次后保存外参矩阵，启动时直接加载，无需每次手动操作。

**C. 实现预存位姿方案**

```python
class AutoRegistration:
    """自动配准：预存位姿 + 精配准验证"""
    
    def __init__(self, config_path: str):
        self.preset_pose = self._load_preset_pose(config_path)
    
    def register(self, current_cloud: np.ndarray, map_cloud: np.ndarray):
        # 1. 使用预存位姿作为初始值
        initial_transform = self.preset_pose
        
        # 2. 用 ICP 做精配准（小范围优化）
        refined_transform = icp_refine(
            source=current_cloud,
            target=map_cloud,
            initial_guess=initial_transform,
            max_iterations=30,
            tolerance=0.01
        )
        
        # 3. 检查配准质量
        fitness = evaluate_registration(refined_transform, current_cloud, map_cloud)
        if fitness > MIN_FITNESS:
            return refined_transform
        else:
            self.get_logger().warn("配准质量不佳，请检查雷达位置")
            return initial_transform
```

## 附录 A：雷达评分公式速查

### 评分公式

$$
\text{Score} = \frac{\text{accurate\_marking\_time} \times 1 + \text{UAV\_counter} \times 20 + \text{signal\_wave\_level} \times 200}{\text{total\_match\_time}}
$$

### 评分要素分解

| 要素 | 符号 | 权重 | 说明 | 对应开发项 |
|------|------|------|------|-----------|
| 精准标记时间 | `accurate_marking_time` | ×1 | 累计精准标记的秒数 | 检测+EKF+标记逻辑 |
| 空中反制次数 | `UAV_counter` | ×20 | 成功反制空中机器人的次数 | 激光反制模块 |
| 信息波等级 | `signal_wave_level` | ×200 | 解析成功的最高难度等级 | 信息波解析模块 |
| 比赛总时长 | `total_match_time` | 分母 | 比赛总时间（秒） | — |

### 评分优化策略

```
ROI 分析（假设 total_match_time = 420s）:

1. accurate_marking_time: 最大 420s → 最大贡献 420/420 = 1.0
   → 需要持续、准确的标记，是基本盘
   → 每提升 1s 准确标记 ≈ +0.24% 得分

2. UAV_counter: 每次 +20 → 每次贡献 20/420 ≈ 4.8%
   → 高收益单次事件
   → 假设成功 3 次 = 60/420 ≈ 14.3%

3. signal_wave_level: 每级 +200 → 每级贡献 200/420 ≈ 47.6%
   → 单项权重最高！
   → 初级解析成功 = 47.6%
   → 中级解析成功 = 95.2%（两级累计）
   → 高级解析成功 = 142.9%（三级累计）

结论：signal_wave_level 是拉开差距的关键项！
      但 accurate_marking_time 是基础，不能放弃。
```

### 被标记进度 P 值机制

```
P 值范围: 0 ~ 150
- 精准标记: P += 5/s（准确识别敌方位置时）
- 错误标记: P -= 10/s（标记错误位置时）
- 无标记:   P 自然衰减

阈值效果:
- P ≥ 100: 触发"易伤"（被标记机器人受伤增加）
- 配合双倍易伤: 伤害增益翻倍

双倍易伤:
- 每场比赛 2 次机会
- 每次持续 30 秒
- 需手动或自动触发
```
